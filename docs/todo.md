먼저 확정된 결정들
아키텍처는 로그인 서버와 API 백엔드를 물리적으로 분리하는 마이크로서비스 구조예요. 인증 방식은 SDK 방식이라, iOS 앱이 구글 로그인으로 ID token을 받아 로그인 서버로 보내고, 로그인 서버가 그걸 검증해 우리 JWT를 발급해요. JWT 서명은 분리 구조에 맞게 RSA 비대칭키(로그인 서버가 개인키로 발급, API 서버가 공개키로 검증)로 가고요. 검증·발급 도구는 스프링 시큐리티 OAuth2 Resource Server(B 방식)로 통일했어요.
앞으로 할 일
크게 보면 로그인 서버 쪽 네 가지와, API 백엔드 쪽 인가 처리, 그리고 그 출발점인 Member 도메인이에요. 순서대로 정리하면 이래요.
0번, Member 도메인 설계. 이게 모든 것의 출발이에요. 구글 sub·email·이름 같은 인증 정보와 우리 서비스 정보(가입일, 역할 등)를 어떻게 담을지, 그리고 지금까지처럼 도메인/엔티티를 분리할지 정해요. JWT에 뭘 담을지, API 서버가 인가를 어떻게 할지가 다 여기서 갈려요.
1번, 구글 ID token 검증. 로그인 서버에서 구글 라이브러리로 앱이 보낸 토큰의 서명·만료·audience를 검증하고, 안의 사용자 정보를 꺼내요.
2번, Member 등록/조회. 검증된 구글 정보로 기존 회원이면 조회, 신규면 등록해요. (0번 도메인 위에서 동작)
3번, JWT 발급. JwtEncoder로 RSA 개인키 서명한 우리 JWT를 발급해요. 이때 Member 식별자를 클레임에 담아서 API 서버가 인가에 쓸 수 있게 하고요.
4번, JWT 검증 (API 백엔드). OAuth2 Resource Server가 RSA 공개키로 들어온 JWT를 검증하고, SecurityContext에 인증 정보를 채워요.
5번, 인가 처리 (API 백엔드). Journey·CityVisit·Post 서비스에서 "이 리소스가 현재 Member의 것인가" 소유권을 검증해요. 이건 라이브러리가 아니라 서비스 코드로요.
여기에 더해, 분리 구조라 RSA 키를 어떻게 관리하고 공유할지(공개키 직접 설정 vs JWK 엔드포인트)도 1~4번 사이 어딘가에서 정해야 해요. 그리고 refresh token을 둘지(access token만으로 시작할지)도 3번 즈음에 결정할 거리로 남아 있어요.
제 생각
0번 Member 도메인부터 시작하는 게 맞아요. 1~5번이 다 Member를 전제로 동작하거든요. Member가 어떤 모양인지, JWT에 뭘 담을지가 정해져야 나머지가 술술 풀려요.
이 정리가 강옥 님이 머릿속에 그린 거랑 맞나요? 혹시 빠지거나 순서를 바꾸고 싶은 게 있으면 조정하고, 맞으면 0번 Member 도메인부터 들어가요.
